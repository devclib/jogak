# Why Jogak

Storybook works, but its stack is heavy: a manager UI shipped as a separate React app, a preview iframe with its own bundler runtime, an addon loader, react-docgen at runtime, a webpack-or-vite-but-both-via-config builder. The catalog itself is small; the surrounding machinery is large.

Jogak takes the opposite stance — strip the machinery, keep the catalog.

## The four levers

### 1. Single Vite instance
No separate manager app. No preview iframe with a second bundler. The same Vite serves the chrome UI and the user content. (User components do mount in an iframe, but it shares the host's Vite scope — see [previewIsolation](/docs/configuration#previewisolation).)

### 2. Lazy virtual modules
The index module exposes only entry **metadata** (title, prop types, default args). Each entry's component is dynamically imported when the user clicks it. **On first dev page load, the module graph contains zero user modules** — idle RSS barely scales with catalog size.

| Catalog size | Storybook idle RSS | Jogak idle RSS |
|---|---|---|
| 100 entries | 403 MB | **321 MB** |
| 500 entries | 489 MB | **345 MB** |

### 3. child_process-isolated ts-morph
Props extraction runs in a separate child process. After 5 seconds of idle the child is `SIGTERM`-ed and the OS reclaims the memory immediately. The main process never carries the full ts-morph type checker.

### 4. In-place HMR
When you edit a `*.jogak.tsx` file, the index metadata is patched via a `ws` custom event, while args/component changes are picked up by the entry virtual module's `self-accept` and rehydrated automatically. **No full reload.** HMR warm median: **153 ms** (size 100) / **199 ms** (size 500).

## What you give up

- **No addon ecosystem.** Jogak intentionally doesn't ship an addon API. If you need a feature, file an issue — we either build it in or document a workaround. (See [Roadmap](/docs/roadmap).)
- **No manager UI customization beyond theme.** Chrome layout is fixed; per-component controls and source view are generated, not configured.
- **No Storybook stories.** Jogak uses `*.jogak.{ts,tsx}` with its own schema. The shapes are similar (`title`, `args`, `argTypes`) but Storybook's `.stories.tsx` files won't be picked up.

If those tradeoffs don't fit, Storybook 8 remains an excellent choice — Jogak is **drop-in** but not a complete superset.

## The name

> **조각** (jogak) is Korean for "**piece**".
> _조각들이 모여 하나의 조각을 만든다_ — Small pieces gather to form one whole.

A component library is made of pieces. The viewer should be too — small, swappable, replaceable. That's the bias behind every design choice on this page.
