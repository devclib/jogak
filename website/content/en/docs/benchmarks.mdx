# Benchmarks

Numbers vs Storybook 8 (Vite builder). Same React catalog (N components, 3 entries each) wired up to both.

## Scaling table

| Metric | size 100 | size 500 | Notes |
|---|---|---|---|
| Dev cold start | **1.7 s** vs 3.3 s (**2.0×**) | **2.0 s** vs 3.6 s (**1.8×**) | "First HTTP 200" |
| Build time | **2.0 s** vs 2.9 s (1.4×) | **4.6 s** vs 5.7 s (1.2×) | Gap widens with catalog size |
| Bundle (gzip) | **108 KB** vs 716 KB (**6.6×**) | **282 KB** vs 1.09 MB (**3.9×**) | Play/Themes/MDX addons added since 1.0 |
| dist total | **340 KB** vs 3 MB (**8.9×**) | **1.58 MB** vs 4.34 MB (2.7×) | Static hosting transfer |
| Idle RSS (dev) | **321 MB** vs 403 MB (1.3×) | **345 MB** vs 489 MB (**1.4×**) | Dev tree RSS median (last measured 1.0) |
| HMR (warm median) | **153 ms** | **199 ms** | args edit → DOM, < 200 ms |

## Reproduce locally

All measurement code lives in [`benchmarks/`](https://github.com/devclib/jogak/tree/main/benchmarks).

```bash
pnpm bench:scale:full   # size 5 / 50 / 100 / 500
pnpm bench:rss          # idle RSS (jogak vs storybook)
pnpm bench:hmr          # HMR latency (size 50, 10 runs)
pnpm bench:baseline     # jogak vs storybook (size 5)
```

These numbers are observed on a recent MacBook Pro. Your absolute numbers will differ; the **ratios** are what we track for regressions.

## What "lightweight" means here

The bundle/RSS ratios come from architectural decisions, not from compression tricks:

- **No manager bundle** — chrome UI ships with the same Vite as user content
- **Zero user modules on first paint** — lazy virtual modules postpone component graph hydration
- **No runtime react-docgen** — ts-morph extracts at build, child process disposed after 5 s
- **Preact (~3 KB) for Web Components** — React runtime never reaches the WC adapter path

See [Why Jogak](/en/docs/why) for the architectural rationale.
